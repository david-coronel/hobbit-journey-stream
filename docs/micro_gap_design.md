# Diseño de Micro Gaps: Bloques Narrativos y Frecuencia de Update

Cada micro gap (≤ 3h) se descompone en **bloques narrativos** de 15–60 minutos.  
Cada bloque tiene un **modo de update** que define cómo avanza el texto en tiempo real.

## Modos de Update

| Modo | Uso | Frecuencia | Ejemplo de contenido |
|------|-----|------------|----------------------|
| **Chat** | Diálogos, monólogos, conversaciones | Cada 2–5 min | Turno de habla, reacción, interrupción |
| **Evento** | Acción puntual, transición, descubrimiento | Cada 5–10 min | "Bilbo tropieza. Dwalin lo ayuda." |
| **Ambiental** | Estado continuo sin cambio drástico | Cada 15–30 min | "El viento aulla entre las rocas." |

## Implicaciones para el Motor Procedural

1. **Un micro gap no es un único bloque.** Son 2–4 bloques consecutivos, cada uno con su propio modo y ritmo.
2. **El modo Chat no es un stream de chat continuo.** Es un turno de habla cada 2–5 minutos. Un diálogo de 20 minutos genera ~4–6 updates, no 20.
3. **El modo Ambiental permite "silencios narrativos"** donde el update es breve: 10–15 palabras que describen continuidad.
4. **Cada bloque puede tener un personaje focal.** No todos los 15 enanos necesitan aparecer. Un bloque puede ser "Dwalin + Balin", otro "Bilbo solo", otro "Thorin observando".
5. **La transición entre bloques es un Evento.** Siempre hay una pequeña acción que cierra uno y abre otro: alguien se levanta, empieza a llover, llega un mensajero, se apaga el fuego.

## Decisiones Pendientes

1. **¿El motor elige bloques probabilísticamente o determinísticamente?**  
   *Opción A:* Cada micro gap tiene un schedule fijo (como en esta tabla).  
   *Opción B:* El motor elige bloques de una bolsa según ubicación, hora y estado del mundo.

2. **¿Los bloques se generan previamente o just-in-time?**  
   *Opción A:* Generar los ~144 updates de micro gaps con LLM antes de iniciar el stream.  
   *Opción B:* Generar cada update en el momento, usando templates + LLM local.

3. **¿Qué pasa si el espectador entra a mitad de un bloque?**  
   El cliente debería mostrar el estado actual del bloque (texto persistente) más el último update recibido.
